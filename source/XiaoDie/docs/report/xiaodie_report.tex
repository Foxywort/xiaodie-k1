\title{基于进迭时空K1的"小蝶"端侧儿童语音大模型系统}
\author{}
\maketitle

% ============================================================
\begin{abstract}
"小蝶"是一套部署在进迭时空K1 MUSE Pi Pro板卡上的端侧儿童语音故事系统。针对现有儿童故事机内容固化、云端语音助手隐私与网络依赖等不足，本作品在Bianbu Linux平台上完成了从语音输入到语音输出的全链路端侧AI部署。系统以GPIO按钮触发录音，经USB麦克风采集16\,kHz单声道音频，由本地常驻的sherpa-onnx Zipformer int8 ASR守护进程完成离线语音识别；识别文本经BM25轻量检索引擎匹配动画IP知识卡片，注入安全约束与角色设定后，送入基于Qwen2.5-1.5B-Instruct微调并量化为Q4\_K\_M GGUF格式的本地大语言模型，由Ollama框架驱动流式生成儿童故事文本；生成的文本按完整中文句分段，逐段送入本地Chaowen Full TTS C守护进程合成语音，经ffmpeg转码后通过3.5\,mm耳机口播放。Qt6原生界面同步显示儿童可理解的交互状态与故事内容，支持按钮打断与重新发起。

训练阶段，项目收集TinyStories Chinese等开放许可数据集共5\,455条样本，经HTML清洗、安全内容过滤与近重复去重后构建5\,180条SFT样本，在RTX 4060 Ti上以LoRA（$r=16$, $\alpha=32$）方式微调2\,000步，最佳eval loss为1.882。模型合并为FP16 Hugging Face格式后，经llama.cpp转换并量化为Q4\_K\_M GGUF，最终模型体积约986\,MB。端侧部署采用ASR/TTS常驻守护进程与模型预热策略：纯LLM推理首token延迟约2.9\,s，生成速度约3.3\,tokens/s；RAG缓存命中后首token延迟约4.8\,s；LLM与TTS并发运行时生成速度约1.6\,tokens/s，通过句子级流式TTS覆盖等待。系统成本低、功耗有限、数据不出本地，适合智能故事机、儿童陪伴机器人和幼教互动终端等场景。
\end{abstract}

\section{作品概述}
\label{sec:overview}
% ============================================================

\subsection{功能与特性}

"小蝶"系统提供完整的端侧语音交互：儿童长按实体按钮说出故事主题，松开后系统依次完成语音识别、知识增强、故事生成与语音合成，最终通过耳机播放个性化的儿童故事。

系统的主要功能包括：（1）端侧离线语音识别，基于sherpa-onnx Zipformer int8模型，将儿童语音转为文本；（2）动画IP知识增强，通过BM25检索156张IP知识卡片（涵盖小猪佩奇、汪汪队、海底小纵队等），在故事生成时注入角色设定与安全约束；（3）端侧大语言模型故事生成，使用Qwen2.5-1.5B-Instruct经LoRA微调并量化为Q4\_K\_M GGUF后在K1上运行；（4）端侧中文语音合成，采用Chaowen Full TTS实现离线高质量语音输出；（5）Qt6原生界面，提供"正在听""正在想故事""讲故事中"等儿童友好的状态显示；（6）按钮打断机制，在任何阶段均可按下按钮重新开始新的故事。

上述链路无需互联网连接，所有AI推理均在K1板卡本地完成。

\subsection{应用领域}

本系统面向3--7岁学龄前儿童的语音陪伴与启蒙阅读场景。

\textbf{智能故事机。}传统故事机预置固定音频内容，儿童无法根据自身兴趣定制故事。"小蝶"系统通过端侧大模型实时生成故事，儿童只需说出"佩奇和乔治整理玩具"等简短主题，即可获得对应的个性化故事，并以自然语音播放。

\textbf{儿童陪伴机器人。}系统可嵌入到具有外壳和扬声器的硬件产品中，作为儿童陪伴机器人的语音交互核心。端侧部署特性使其无需依赖云端服务器，适合家庭和幼教机构使用。

\textbf{幼教互动终端。}RAG知识库可扩展为科普、安全教育、生活习惯培养等不同主题的卡片集合，由此将系统的应用范围从故事娱乐拓展到教育引导。

\textbf{RISC-V端侧AI应用示范。}本项目验证了在K1 RISC-V CPU上运行1.5B参数大语言模型、离线ASR和离线TTS的可行性，为同类端侧AI产品提供了工程参考。

\subsection{主要技术特点}

\textbf{全链路端侧AI推理。}ASR、LLM与TTS三个AI模块均在K1板卡本地离线运行，不依赖云端API。ASR使用sherpa-onnx C API封装的Zipformer int8模型，LLM使用Ollama驱动的Q4\_K\_M量化GGUF模型，TTS使用sherpa-onnx C API封装的Chaowen Full VITS模型。

\textbf{轻量RAG知识增强。}采用BM25文本检索替代向量embedding模型，在K1端无需额外的embedding推理开销。知识卡片包含角色关系、别名、安全策略等结构化信息，检索时按franchise过滤并强制注入安全类卡片。

\textbf{常驻守护进程架构。}ASR与TTS均以C语言编写的daemon方式运行，程序启动时一次性加载模型，后续交互无需重复加载。ASR daemon首次加载约27\,s，常驻后识别延迟仅取决于录音时长。TTS daemon同样常驻内存，内部分离合成线程与播放线程。

\textbf{句子级流式TTS。}LLM流式生成token时，系统将其按完整中文句子分组（25--90字符/段），每攒齐一段即送入TTS合成。在LLM还在生成后续内容的同时，前面的段落已在合成或播放中，缩短了用户等待时间。

\textbf{领域微调与安全约束。}使用5\,180条开放许可的儿童故事样本对Qwen2.5-1.5B-Instruct做LoRA微调，训练过程中仅对assistant回答计算loss。Modelfile中设定较低的temperature（0.18）和较高的repeat\_penalty（1.16），system prompt要求故事内容安全、禁止输出表情符号与解释性文字。

\subsection{主要创新点}

\begin{enumerate}[leftmargin=*]
    \item 在K1 RISC-V CPU上实现了ASR + RAG + 1.5B LLM + TTS四模块端侧全链路部署，无需网络连接即可完成从语音输入到语音输出的完整交互。
    \item 提出句子级流式TTS策略：LLM逐token生成的同时，完整句子即时送入TTS后台合成与播放，以流水线方式覆盖各模块的串行延迟。
    \item 设计了面向儿童动画IP的轻量BM25 RAG方案，不依赖embedding模型，直接将角色设定与安全约束注入LLM生成过程。
    \item ASR与TTS均采用C语言常驻daemon架构，消除了模型反复加载的冷启动开销，配合模型预热脚本将系统就绪时间控制在可接受范围内。
\end{enumerate}

\subsection{设计流程}

项目开发大致经历了四个阶段。先是数据准备与模型训练——收集TinyStories Chinese等开放数据集并清洗，在RTX 4060 Ti上对Qwen2.5-1.5B-Instruct做LoRA微调，合并权重后量化为Q4\_K\_M GGUF。之后进入端侧模块开发，分别实现了ASR C daemon、TTS C daemon、RAG检索脚本和LLM调用脚本，并在K1板卡上逐模块调试。各模块调通后进行系统集成，编写按钮控制逻辑将各模块串联，开发Qt6界面，实现打断与重新开始机制。最后是性能优化与测试，主要工作包括模型预热、TTS线程优先级调度、prefix缓存等。

各模块之间的协作关系如下：按钮控制脚本（Python）负责管理整个交互状态机，在按钮按下时启动录音，松开时停止录音并将WAV路径通过stdin送入ASR daemon；ASR daemon将识别结果以JSON格式通过stdout返回；按钮脚本解析识别文本后调用RAG + LLM脚本，后者以流式方式将生成的token通过FIFO送入TTS daemon；TTS daemon内部的合成线程与播放线程并行工作，将合成后的PCM音频经ffmpeg和aplay输出到硬件。Qt6前端以独立进程运行，通过读取按钮脚本的stdout获取状态事件。

\subsubsection{电路模块介绍}

本项目未涉及自制PCB电路设计，所有硬件均基于K1 MUSE Pi Pro板卡的原生接口。GPIO按钮电路为最简单的上拉/下拉配置：按钮一端接GPIO\,35引脚，另一端接GND，软件中配置为active low模式，按下时GPIO电平拉低。音频输入输出分别由板载USB Host控制器和板载音频编解码器（ES8326）处理，无需额外的外围电路。

\subsection{软件系统介绍}

\subsubsection{软件整体介绍}

软件系统由五个主要模块组成，运行在Bianbu Linux上：

\begin{enumerate}[leftmargin=*]
    \item \textbf{ASR模块}：C语言编写的常驻daemon（\texttt{xiaodie\_asr\_daemon}），使用sherpa-onnx C API加载Zipformer int8模型，通过stdin接收WAV路径、stdout输出JSON识别结果。
    \item \textbf{LLM/RAG模块}：Python脚本（\texttt{xiaodie\_rag\_ollama\_stream.py}），负责BM25检索IP知识卡片、构造prompt、调用Ollama本地HTTP API进行流式推理，并将完整句子通过FIFO送入TTS。
    \item \textbf{TTS模块}：C语言编写的常驻daemon（\texttt{chaowen\_tts\_daemon}），使用sherpa-onnx offline TTS C API加载Chaowen Full VITS模型，内部包含合成线程和播放线程，通过stdin接收文本、输出合成状态日志。
    \item \textbf{按钮控制模块}：Python脚本（\texttt{xiaodie\_button\_story.py}），管理GPIO按钮状态机，协调录音、ASR、LLM和TTS各环节的启动与打断。
    \item \textbf{Qt6界面}：C++编写的原生GUI（\texttt{xiaodie\_qt\_gui}），读取按钮脚本的stdout事件，翻译为"小蝶正在听你说话""小蝶正在想故事"等儿童友好提示。
\end{enumerate}

\subsubsection{ASR模块}

ASR daemon使用sherpa-onnx提供的streaming Zipformer transducer模型（中英文+标点，int8量化）。模型文件包括encoder.int8.onnx、decoder.onnx、joiner.int8.onnx和tokens.txt，provider设为CPU，解码方式为greedy\_search，线程数为4。

daemon启动后常驻内存。按钮脚本将录音WAV路径通过stdin发送给daemon，daemon读取WAV并执行识别，结果以JSON格式输出：

\begin{lstlisting}[language={}]
{"ok":true, "text":"佩奇和乔治整理玩具",
 "audio_s":3.2, "elapsed_s":1.1, "rtf":0.34}
\end{lstlisting}

ASR模块的关键优化包括：（1）录音结束后修复arecord被SIGINT打断时的WAV header长度字段；（2）在录音尾部补充500\,ms静音padding，提升松开按钮后末尾文字的识别完整度；（3）识别失败时自动重启daemon。

\subsubsection{LLM与RAG模块}

LLM推理通过Ollama框架完成。模型文件为qwen2\_5-1\_5b-xiaodie-story-Q4\_K\_M.gguf（约986\,MB），在Ollama中注册为xiaodie-story-1.5b:latest。Modelfile配置了如下推理参数：上下文长度4096（端侧测试时限制到1024以降低延迟）、temperature\,0.18、top\_p\,0.72、repeat\_penalty\,1.16。system prompt要求输出安全的儿童故事正文，禁止输出emoji和解释性文字。

RAG模块在调用LLM之前执行。输入为ASR识别文本，BM25检索引擎对156张IP知识卡片进行匹配，默认取top\_k=2张卡片（每张限80字符），按franchise过滤后强制加入curated\_fact、safety\_policy和alias\_policy类型的卡片。检索结果注入prompt，明确要求模型"只使用知识卡片中出现的人物、地点、关系和设定"。

BM25的token化函数同时提取英文词、中文单字和中文bigram，以提高中英文别名（如Peppa Pig/小猪佩奇）的匹配率。该方案不依赖embedding模型，在K1端无额外的向量计算开销。

LLM流式推理过程中，每个token经过emoji/ANSI控制字符/Markdown符号过滤后，进入句子分组器。分组器按2--4个完整中文句子组成一个chunk（25--90字符），确保每个TTS输入都是语义完整的段落，不会在半句话中间切断。

\subsubsection{TTS模块}

TTS daemon使用sherpa-onnx offline TTS C API和Chaowen Full中文语音模型。程序启动时加载模型到内存，此后所有合成请求共享同一模型实例。

daemon通过stdin接收文本和控制指令。普通UTF-8文本行追加到待合成缓冲区；\texttt{::flush}强制合成剩余文本；\texttt{::mark TOKEN}在播放队列中插入标记，用于通知UI某段音频已播完；\texttt{::reset}清空待合成文本和任务队列。

daemon内部包含两个工作线程：合成线程将文本转为PCM音频数据，播放线程将PCM经ffmpeg重采样后通过aplay送到硬件音频设备（ES8326编解码器）。两个线程通过队列连接，合成与播放可并行进行。

TTS模块与Qt界面之间通过\texttt{::mark}协议实现同步：LLM每生成完一段文本并送入TTS后，在队列末尾插入一个mark；TTS播放完该段音频后输出mark\_done日志；Qt前端据此显示对应段落的文字，并在所有段落播完后显示"故事讲完了"。

\subsubsection{按钮控制与状态机}

按钮控制脚本管理整个交互的生命周期。状态机的工作流程如下：

\begin{enumerate}[leftmargin=*]
    \item 程序启动时，先启动TTS服务和ASR daemon，等待模型加载完成。
    \item 儿童按住按钮：若当前有未完成的任务，先打断录音、ASR、LLM和TTS；随后使用arecord开始录音。
    \item 儿童松开按钮：发送SIGINT停止arecord，修复WAV header，将WAV路径送入ASR daemon。
    \item ASR返回文本：若文本过短则通过TTS播放提示"没有听清楚，可以再说一遍吗"；若文本有效则进入故事生成。
    \item 再次按下按钮：强制终止当前故事生成进程，重启或清空TTS服务，重新开始录音。
\end{enumerate}

打断机制的实现依赖generation token：每次新的交互分配一个新token，旧线程检测到token变化后停止输出，避免上一轮的残余文本或音频影响新的交互。

\subsubsection{Qt6原生界面}

用户界面采用Qt6 C++编写，避免了Web方案的浏览器开销。界面左侧显示进迭时空logo、标题、"开始使用"和"结束使用"按钮；右侧显示当前状态和故事文本。

Qt进程以子进程方式启动按钮控制脚本，读取其stdout中的事件标签。事件到界面提示的映射示例：\texttt{recording...}$\rightarrow$"小蝶正在听你说话"；\texttt{thinking story}$\rightarrow$"小蝶正在想一个适合你的故事"；\texttt{[xiaodie\_played] 文本}$\rightarrow$显示该段故事文字；\texttt{[xiaodie\_tts\_done]}$\rightarrow$"故事讲完了，还可以继续点播新故事"。所有面向用户的文字均不包含ASR、GPIO、Chaowen Full等技术术语。

\section{总结}
\label{sec:conclusion}
% ============================================================

\subsection{可扩展之处}

系统目前还有几个方向可以进一步拓展。模型方面，项目同时训练并导出了Qwen3-4B的GGUF量化模型（Q4\_K\_M约2.5\,GB），在内存更充裕的板卡或下一代K3芯片上可直接切换到4B模型，故事生成质量会有明显提升。RAG知识库目前包含5个动画IP的156张卡片，后续可扩展更多IP或引入科普、安全教育等非娱乐类知识主题。ASR方面，当前使用的Zipformer模型对儿童语音的识别准确率仍有提升余地，可通过采集儿童语音数据进行针对性微调。TTS方面，若后续出现适配RISC-V的轻量神经网络语音克隆方案，可实现更丰富的角色配音效果。交互层面，目前仅支持单轮对话，可扩展为多轮对话模式以实现更自然的故事续编能力。

\subsection{心得体会}

本项目从选题到调试完成，先后经历了数据准备、模型训练、端侧移植和系统集成。以下记录开发过程中的一些具体体会。

在模型选型阶段，我们原计划使用Qwen3-4B作为端侧模型。4B模型在台式机上生成的故事质量确实更好，句式更丰富，角色描写也更细致。但部署到K1板卡后，4B模型的内存占用和推理延迟都给有限的硬件资源带来了压力，特别是与TTS并发运行时CPU竞争严重。最终我们决定以1.5B作为端侧主力模型，4B则保留为质量对照。实际使用中发现，端侧场景下模型选型不能只看生成质量，还要考虑加载时间、推理速度以及和其他模块的CPU竞争。

数据清洗是另一个投入大量时间的环节。开放数据集中混杂了相当比例的教学类文本、过短或过长的片段以及包含不适宜儿童内容的样本。我们编写了多级过滤规则，包括中文字符比例检测、关键词安全过滤和中文shingle近重复去重。经过清洗后保留的5\,455条样本虽然在总量上不算大，但质量比较齐整，训练出的模型在儿童故事场景下表现尚可。

ASR和TTS改成daemon常驻后，端侧体验改善了不少。改造前每次交互都要重新加载模型，ASR加载超过27秒，TTS也要数秒初始化，实际使用时等待感很强。改为常驻daemon后，模型只在应用启动时加载一次，后续录音识别和语音合成均在秒级完成。daemon化在嵌入式开发中算不上新做法，但在端侧AI场景下确实必要。

句子级流式TTS是应对K1算力限制的主要手段。Chaowen Full的RTF大于1，若等LLM生成完整故事再统一合成，等待时间过长。把LLM输出按完整句子分段后即时送入TTS后台，播放前面段落的同时LLM继续生成后续内容，用户实际感受到的等待就短了很多。

Qt界面开发时我们有意避开了所有技术术语。面向幼儿的产品不该出现"ASR""TTS""GPIO"之类的词，换成"小蝶正在听你说话""小蝶正在想故事"这样的说法更合适。文字和语音的同步也花了不少功夫——TTS播放完某段音频后界面才显示对应文字，否则文字先于声音出现会让人觉得不协调。

做完这个项目后，我们对RISC-V平台跑AI应用有了比较具体的感受。K1本身不是为大模型推理设计的芯片，但通过模型量化、常驻进程、prefix缓存和流式处理这几项工程优化，在端侧跑通包含ASR、LLM和TTS的完整语音链路是可行的。记录下这些踩坑经验，如果后续有人在RISC-V上做类似的端侧AI应用，或许能少走一些弯路。
